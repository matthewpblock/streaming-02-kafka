## Custom Project (Garmin Data)

### Dataset

For this custom implementation, the data source was entirely changed from the original sales data to a personal health and fitness dataset.

- **File Name:** `GarminTrainingData.csv`
- **Record Types:** It contains individual fitness activity records (e.g., running, walking, cycling) logged by a Garmin wearable device.
- **Key Fields Included:** The dataset features various fitness metrics, primarily focusing on `Type`, `Distance (km)`, and `Duration (min)`.
- **Dataset Modification:** The dataset was sourced externally and completely replaces the generic `sales.csv` example to demonstrate real-world fitness tracking.

### Kafka Messages

The messaging pipeline was tailored to handle streaming fitness activities.

- **Producer Output:** The producer reads rows from the Garmin CSV and sends each activity as a JSON-formatted dictionary to Kafka.
- **Topic:** `streaming-02-kafka-garmin`
- **Message Key:** The key was changed to use the `Type` field (e.g., "running", "walking"). This allows activities of the same type to be routed to the same partition, preserving order for specific workout types.
- **Message Fields:** The raw fields remain identical to the CSV headers, but the consumer enriches them downstream.

### Consumer Processing

The consumer is designed to ingest and augment the raw activity data in real-time.

- **Reception:** Receives individual Garmin activity JSON messages from the `streaming-02-kafka-garmin` topic.
- **Consumption Limit:** Consumes up to 1000 messages per run (or until a 10.0s timeout is reached without new messages).
- **Logging:** Logs the receipt of each message, the original raw dictionary, and the ongoing count of consumed messages.
- **Output:** Writes the successfully processed records to `data/output/consumed_garmin_data.csv`.
- **Processing/Filtering:** The consumer dynamically calculates a new derived metric—`Speed (km/h)`—by extracting `Distance (km)` and `Duration (min)` from the incoming message, applying the necessary math, and injecting it into the row before saving.

### Experiments

Several technical modifications were implemented to customize the data flow:

- **Phase 4 Changes:** Altered the `.env` settings to use a custom topic (`streaming-02-kafka-garmin`) and a custom group ID (`streaming-consumer-group-garmin`). The `PRODUCER_MESSAGE_COUNT` was increased to 10, and the `PRODUCER_MESSAGE_INTERVAL_SECONDS` was reduced to 1.5s for a smoother streaming visual.
- **Phase 5 Application:** Swapped the data domain entirely to Garmin activity data. Updated the Kafka message key logic to route by activity `Type`, and implemented a live analytics derivation (`Speed (km/h)`) directly within the consumer processing loop.

### Results

Upon execution, the producer successfully launched, read from `GarminTrainingData.csv`, and streamed 10 messages spaced 1.5 seconds apart into the custom topic. The consumer, running concurrently, instantly detected the messages, generated the new `Speed (km/h)` attribute for each, logged the updates to the console, and safely wrote the enriched dataset into `consumed_garmin_data.csv`. Once the producer finished, the consumer safely timed out and shut down cleanly.

### Interpretation

This exercise demonstrated the power and flexibility of Apache Kafka as a decoupling buffer and real-time processing engine.

- **What Changed:** We transitioned from a static retail scenario into an active Internet of Things (IoT) wearable device stream.
- **Lessons Learned:** Watching the messages move through Kafka highlighted how producers and consumers operate completely independently. The consumer doesn't need to know how the Garmin data was generated, only how to react to it.
- **Business Value:** In a real-world scenario, this pipeline could inform a live health dashboard. A fitness app or health insurance provider could use this exact stream to monitor user activity levels as they happen, triggering alerts or achievements when specific speed or distance thresholds are met.
- **Business Intelligence Gained:** By deriving `Speed (km/h)` on the fly, the organization instantly gains insight into user performance and exertion levels without requiring downstream batch processing, enabling real-time leaderboards or coaching feedback.
